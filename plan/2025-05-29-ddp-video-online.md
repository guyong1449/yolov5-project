## 视频推理的多卡方案全景

### 一、离线批处理（你当前的方案）

```
┌─────────────────────────────────────────────────┐
│  Video File (1000 frames)                       │
│  ├─ rank 0: frames 0-249                        │
│  ├─ rank 1: frames 250-499                      │
│  ├─ rank 2: frames 500-749                      │
│  └─ rank 3: frames 750-999                      │
│                                                  │
│  各 rank 独立推理 → rank 0 汇总结果              │
└─────────────────────────────────────────────────┘
```

| 项目 | 说明 |
|------|------|
| 输入 | 已录制好的视频文件 |
| 分片方式 | 按帧号范围静态分片 |
| 延迟 | 视频越长越久，但总吞吐高 |
| 通信开销 | 仅最后汇总，几乎无 |
| 适用 | 视频分析、离线标注、批量处理 |

---

### 二、在线方案 A：多路独立流（最简单）

```
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ Camera 0 │  │ Camera 1 │  │ Camera 2 │  │ Camera 3 │
│   ↓      │  │   ↓      │  │   ↓      │  │   ↓      │
│  rank 0  │  │  rank 1  │  │  rank 2  │  │  rank 3  │
│ (NPU:0)  │  │ (NPU:1)  │  │ (NPU:2)  │  │ (NPU:3)  │
└──────────┘  └──────────┘  └──────────┘  └──────────┘
  独立推理       独立推理       独立推理       独立推理
```

| 项目 | 说明 |
|------|------|
| 输入 | 4 路独立摄像头/直播流 |
| 分片方式 | 按流分配，每路占一张卡 |
| 延迟 | 最低，每帧独立处理 |
| 通信开销 | 零，各 rank 完全独立 |
| 扩展性 | 线性，加卡 = 加路数 |
| 限制 | 要求流数 ≥ 卡数，否则卡闲置 |
| 适用 | 安防监控、多机位直播、工厂多产线检测 |

**实现复杂度**: 低。本质上是单卡推理的 N 副本，不需要 DDP 的 allreduce。

---

### 三、在线方案 B：帧级动态分配（单流高吞吐）

```
┌─────────────┐
│ Camera/流    │
│   ↓ ↓ ↓ ↓   │
│  Dispatcher │  ← 帧到达时分配给空闲 rank
│  ↙  ↓  ↓  ↘ │
│ r0 r1 r2 r3 │  ← 每个 rank 独立处理一帧
│  ↘  ↓  ↓  ↙ │
│  Aggregator │  ← 按原始顺序重组结果
└─────────────┘
```

| 项目 | 说明 |
|------|------|
| 输入 | 单路视频流 |
| 分片方式 | 帧级动态负载均衡 |
| 延迟 | 中等（需等最慢的 micro-batch） |
| 通信开销 | 低（帧分发 + 结果收集） |
| 扩展性 | 接近线性，但有调度开销 |
| 限制 | 输出顺序可能乱序，需要重排 |
| 适用 | 单路高帧率视频（如 120fps 检测） |

**实现复杂度**: 中。需要一个 dispatcher 进程 + 结果重排逻辑。

---

### 四、在线方案 C：流水线并行（单流低延迟）

```
┌────────────────────────────────────────────┐
│  rank 0: 预处理(解码/resize)               │
│    ↓                                       │
│  rank 1: 模型推理(前向)                     │
│    ↓                                       │
│  rank 2: 后处理(NMS/绘框)                  │
│    ↓                                       │
│  rank 3: 编码/推流                          │
└────────────────────────────────────────────┘
```

| 项目 | 说明 |
|------|------|
| 输入 | 单路视频流 |
| 分片方式 | 按处理阶段分配 |
| 延迟 | 单帧最低（各阶段并行） |
| 通信开销 | 高（每帧在 rank 间传递） |
| 扩展性 | 差（瓶颈在最慢的阶段） |
| 限制 | 各阶段耗时需均衡，否则某 rank 空等 |
| 适用 | 对单帧延迟极敏感的场景（自动驾驶、机器人） |

**实现复杂度**: 高。需要 rank 间通信协议、缓冲区管理、背压控制。

---

### 五、在线方案 D：批缓冲（折中方案）

```
┌───────────────────────────────────────┐
│  帧到达 → 积攒 N 帧(如 4 帧)          │
│    ↓                                  │
│  micro-batch 分给 4 个 rank           │
│    rank 0: frame[0]                   │
│    rank 1: frame[1]                   │
│    rank 2: frame[2]                   │
│    rank 3: frame[3]                   │
│    ↓                                  │
│  聚合 → 输出 → 积攒下一批              │
└───────────────────────────────────────┘
```

| 项目 | 说明 |
|------|------|
| 输入 | 单路视频流 |
| 分片方式 | 按 micro-batch 分片 |
| 延迟 | 可控（batch_size=4 → 延迟 4 帧） |
| 通信开销 | 低 |
| 扩展性 | 好，batch_size 可调 |
| 限制 | 延迟 = batch_size × 帧间隔 |
| 适用 | 可容忍少量延迟的实时场景（视频会议、直播标注） |

**实现复杂度**: 低-中。在你现有 DDP 代码基础上加一个帧缓冲层即可。

---

### 六、混合方案：离线 + 在线共存

```
┌─────────────────────────────────────────┐
│  同一套代码，两种运行模式                │
│                                          │
│  --mode batch  → 离线分片（当前方案）    │
│  --mode online → 在线模式（选 A/B/C/D） │
│                                          │
│  共享：模型加载、后处理逻辑、日志格式    │
└─────────────────────────────────────────┘
```

---

### 决策矩阵

| 你的情况 | 推荐方案 | 理由 |
|----------|----------|------|
| 多路摄像头，每路独立 | A（多路独立流） | 最简单，零通信，线性扩展 |
| 单路高帧率，追求吞吐 | B（帧级动态分配） | 充分利用多卡，接近线性加速 |
| 单路，极低延迟要求 | C（流水线并行） | 单帧延迟最低，但实现复杂 |
| 单路，可容忍几十ms延迟 | D（批缓冲） | 改动最小，在现有代码上加缓冲层 |
| 既有离线又有在线 | 混合方案 | 一套代码两种模式 |

---

### 关于 ddp-path-verification skill 的影响

如果你选择在线方案 A（多路独立流），那这个 skill 的核心假设需要调整：

- 当前 skill 假设：rank 之间需要 allreduce（梯度同步或结果聚合）
- 方案 A：rank 之间完全独立，不需要任何通信
- skill 的 "work partitioning" 和 "aggregation" 检查点在这种模式下不适用

如果你选择方案 B/D，现有 skill 的框架仍然适用，只需要把"按帧范围分片"改成"按帧号 mod world_size 分片"。
